# Repository Guidelines

## Project Structure & Module Organization
- `src/` contains the NestJS application code. Entry point is `src/main.ts`.
- `src/app.module.ts` is the root module that wires feature modules.
- Feature modules follow `*.module.ts` naming (example: `src/health.module.ts`, `src/orders.module.ts`).
- Controllers and services follow feature-based names (example: `src/health.controller.ts`, `src/health.service.ts`).
- `test/` contains end-to-end tests and Jest config for e2e (`test/jest-e2e.json`).
- `dist/` is the build output generated by `npm run build`.

## Build, Test, and Development Commands
- `npm run start` starts the app using the Nest CLI.
- `npm run start:dev` runs in watch mode for local development.
- `npm run start:debug` runs with the Node inspector attached.
- `npm run build` compiles TypeScript into `dist/`.
- `npm run start:prod` runs the compiled app from `dist/main`.
- `npm run lint` runs ESLint (with auto-fix) across `src/` and `test/`.
- `npm run format` formats TypeScript files with Prettier.
- `npm run test` runs unit tests (Jest) in `src/`.
- `npm run test:watch` runs unit tests in watch mode.
- `npm run test:cov` runs tests with coverage.
- `npm run test:e2e` runs end-to-end tests from `test/`.

## Coding Style & Naming Conventions
- Language: TypeScript with NestJS conventions.
- Formatting: Prettier is the source of truth. Use `npm run format` before committing.
- Linting: ESLint is configured via `eslint.config.mjs`. Fix issues via `npm run lint`.
- Tests: unit test files use `*.spec.ts` (example: `src/health.controller.spec.ts`). E2E tests use `*.e2e-spec.ts` in `test/`.

## Testing Guidelines
- Framework: Jest (unit and e2e).
- Unit tests live in `src/` alongside code; e2e tests live in `test/`.
- Use `npm run test:cov` to check coverage locally.

## Commit & Pull Request Guidelines
- This repository currently has no Git commit history, so there are no established commit message conventions yet.
- Suggested convention: short, imperative summaries (e.g., "Add orders endpoint").
- PRs should include a concise description, testing notes (commands run), and any relevant context or screenshots if behavior changes.

## Configuration & Secrets
- No configuration files are defined in this repo yet. If adding environment variables, document them in the PR and avoid committing secrets.
